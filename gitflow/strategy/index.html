Branching and Workflow Strategy

Our development workflow follows a Git-flow model, structured for efficient feature development, testing, and deployment. Below is an explanation of the purpose of each branch, its source, and how it fits into the CI/CD process.

Branch and Source Mapping:

  1. main branch:
    • Source: No source; this is the primary branch that everything else eventually flows into. It represents production-ready code.
    • Purpose:
      • This branch should only contain code that is ready for production deployment.
      • Direct commits should never be made here.
      • Code merged into main has passed all testing phases and is ready for deployment.
      • Tags for production releases are applied to this branch.
    git checkout main
  1. develop branch:
    • Source: main
    • Purpose:
      • The develop branch is where all feature development branches are merged after testing.
      • It serves as the integration branch before releasing any feature into production.
      • No direct commits should be made here; instead, work is merged into develop via feature branches.
    git checkout -b develop main
  1. RT-{feature-name} (Feature branches):
    • Source: develop
    • Naming Convention: RT-{feature-name} (e.g., RT-login-system)
    • Purpose:
      • Feature branches are created off develop for specific feature development.
      • Once feature work is complete and tested, it is merged back into develop via a pull request.
      • Each feature branch should follow the naming convention RT-{feature-name} (e.g., RT-Server for server-related features).
    git checkout -b RT-{feature-name} develop
  1. release/* branch:
    • Source: develop
    • Naming Convention: release/v{version-number} (e.g., release/v1.1.0)
    • Purpose:
      • Release branches are created when preparing for a new version of the product.
      • This branch is used for final testing and minor bug fixes before production release.
      • After testing, the release branch is merged into both main and develop, ensuring that both branches are updated with the latest release.
      • Tags representing the version are applied once the release is finalized.
    git checkout -b release/v1.1.0 develop
  1. hotfix/* branch:
    • Source: main
    • Naming Convention: hotfix/v{version-number} (e.g., hotfix/v1.0.1)
    • Purpose:
      • Hotfix branches are created directly from main to address urgent production issues.
      • Once the fix is applied, it is merged into both main and develop, ensuring that the hotfix is reflected in both the production and development versions of the code.
      • Tags representing the fix version are applied.
    git checkout -b hotfix/v1.0.1 main

Development Workflow Summary:

  • Feature Development: Developers create feature branches off develop using the naming convention RT-{feature-name}. These branches are dedicated to specific features, such as RT-Server, and are merged back into develop via pull requests after testing.
  • Integration & Testing: The develop branch serves as the integration branch, where all feature branches are merged. This branch allows for continuous integration (CI) testing. The CI/CD pipeline should run builds, unit tests, and code quality checks (e.g., using clang-format).
  • Release Preparation: When a new version is ready to be tested, a release branch (e.g., release/v1.1.0) is created from develop. This branch is used for final testing and bug fixes. Once the release is stable, it is merged into both main and develop and tagged (e.g., v1.1.0).
  • Hotfixes: Urgent production fixes require creating a hotfix branch directly from main (e.g., hotfix/v1.0.1). After applying the fix, it is merged back into main and develop to maintain consistency across the codebase.

Branching Structure:

BranchSourcePurpose
mainNoneProduction-ready code.
developmainIntegration of all feature branches, bug fixes, and updates.
RT-{feature-name}developNew feature development.
release/*developFinal release preparation and bug fixes for a new version.
hotfix/*mainUrgent fixes for production issues.

By following this branching and workflow strategy, we ensure a well-organized development process, where features are developed in isolation, tested thoroughly, and integrated smoothly into the final production code. This allows for better collaboration, cleaner code management, and smoother CI/CD automation.

Pre-Push Checklist

Before pushing your code to any branch, it’s crucial to ensure that your changes are properly prepared and compliant with the project’s standards. Follow these steps to ensure a smooth workflow:

  1. Code Quality and Formatting:
    • Run Code Linters: Use tools like clang-tidy or cppcheck to check for code quality issues.
    • Format Code: Ensure your code adheres to the project’s formatting guidelines. Run clang-format or the relevant formatter for your language to maintain consistency.
    clang-format -i **/*.cpp **/*.hpp
  1. Unit Testing:
    • Write and Run Unit Tests: Execute all relevant unit tests to confirm that your changes do not break existing functionality.
    ctest
    • Check Test Results: Verify that all tests pass. If any tests fail, resolve the issues before proceeding.
  1. Build Verification:
    • Build the Project: Ensure that the project builds successfully with your changes.
    cmake . && make
    • Verify Build Output: Check that the build output is as expected and that no errors occur during the build process.
  1. Code Review:
    • Self-Review: Review your code changes for any potential issues, such as logic errors, incomplete implementations, or commented-out code that should be removed.
    • Prepare for Peer Review: Ensure your code is ready for review by others. This includes writing clear and concise commit messages.
  1. Documentation:
    • Update Documentation: If your changes impact the project’s documentation, update it accordingly. This could include code comments, README files, or other related documents.
  1. Commit and Push:
    • Commit Changes: Use meaningful commit messages that clearly describe the changes you have made.
    git add .
git commit -m "Describe the changes here"
    • Push Changes: Push your changes to the appropriate branch (e.g., feature branch, develop, or release).
    git push origin your-branch-name
  1. Open Pull Request:
    • Create Pull Request: If you are working on a feature branch or bug fix branch, create a pull request targeting the develop or main branch, depending on your workflow.
    • Assign Reviewers: Add relevant team members as reviewers and provide a clear description of what has been changed and any specific areas they should focus on.
  1. Verify CI/CD Pipeline:
    • Check CI/CD Status: After pushing, monitor the CI/CD pipeline to ensure that the build and test processes run successfully.
    • Address Issues: If the pipeline reports any issues, address them promptly.

Summary of Pre-Push Steps

  1. Code Quality and Formatting: Ensure code is clean and formatted.
  1. Unit Testing: Run and verify unit tests.
  1. Build Verification: Confirm the project builds successfully.
  1. Documentation: Update any impacted documentation.
  1. Commit and Push: Commit with meaningful messages and push changes.
  1. Open Pull Request: Create a PR for review and merge.
  1. Verify CI/CD Pipeline: Check and address any pipeline issues.

Following this pre-push checklist will help maintain high-quality code and smooth integration into the main branches. Let me know if you need any additional details or adjustments!
